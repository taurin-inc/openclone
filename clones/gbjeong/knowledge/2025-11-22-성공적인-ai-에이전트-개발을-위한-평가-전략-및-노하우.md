AI Agent 만들 때 LLM API 만 연결하면 된다고요? 한 번도 evaluation을 안 해본 분일겁니다.. 제대로 만들기 위해서는 ML 모델링하듯 평가를 접근해야 합니다. 이런 관점이 일반적인 머신러닝(ML)을 할 때의 경험이 큰 도움이 됩니다. 제가 경험하며 정리한, 특히 Vertical Domain Agent의 평가 팁은 이렇습니다.

1. LLM as a Judge: 기본기부터 확실하게, 이것만 해도 절반은 갑니다.
일반적으로 다음과 같은 지표들을 사용합니다. 다행인 점은, 이런 일반적인 지표들은 복잡한 프롬프트 엔지니어링 없이도 LLM이 꽤 잘 평가해줍니다.

- Conciseness: 답변이 간결하고 군더더기가 없는가?
- Relevance: 질문의 의도에 맞는 답변인가?
- Correctness: 사실 관계가 정확한가?
- Coherence: 답변의 논리가 일관적인가?
- Harmfulness: 유해한 내용은 없는가?
- Criminality: 범죄를 조장하지는 않는가?

2. Training 셋과 Test 셋의 분리: Agent도 '과적합'된다

"학습(Training)을 시키는 것도 아닌데 Training/Test 셋이 왜 필요해?"라고 생각하실 수 있습니다. 하지만 Agent 개발에서도 이 개념은 매우 중요합니다.

우리는 Agent의 성능을 높이기 위해 시스템 프롬프트를 깎고, Few-shot 예제를 추가하고, RAG/Tool 파이프라인을 수정합니다. 만약 준비된 데이터셋(Training Set)에만 맞춰서 프롬프트를 최적화하다 보면, 시스템 프롬프트가 해당 데이터에만 과적합(Overfitting)되는 현상이 발생합니다. 이런 역학이 ML에서의 '학습'과 유사합니다. 

결국 "우리 evaluation set에서는 다 맞췄어!"라고 자신했지만, 실제 유저들의 질문에서는 엉뚱한 대답을 하는 경우가 생깁니다.

저는 문제를 쉽게 풀기 위해서 이렇게 합니다.
    1.  일단 Training 셋에 대해 오버피팅을 시킵니다. (Test 셋에서는 성능이 떨어짐)
    2.  그 후, Test 셋과의 성능 격차를 줄이기 위해 요소를 하나씩 제거하거나 일반화합니다.

3. Garbage In, Garbage Out: 도메인 엑스퍼트와 함께 만드는 Q-A 셋의 중요성

평가의 핵심은 결국 데이터입니다. 제품의 목적에 딱 맞는 Q-A(질문-답변) 셋을 구성하는 것이 무엇보다 중요합니다.

특히 답변(Ground Truth)의 기준이 모호하면 안 됩니다. "적당히 좋은 답변" 같은 건 없습니다. 기준이 모호하다는 건 제품의 방향성이 명확하지 않다는 뜻이기도 합니다. Vertical Domain에서 이렇게 접근하면 성능을 절대 만족시킬 수 없습니다.

또한, 모든 질문에 답하려고 하지 마세요.
- 유저의 질문이 들어왔을 때, "이게 우리 Agent가 답변할 질문인가?"를 먼저 판단하는 가드레일이 필요합니다.
- 악성 케이스를 막겠다고 검증 로직을 무한정 늘리기보다는, 목적에 따라 기능을 쪼개서 평가하는 것이 성능 유지에 훨씬 유리합니다.

4. 결국은 도메인 이해도 (Domain Knowledge)

마지막으로 강조하고 싶은 건 도메인에 대한 깊은 이해입니다.
프로젝트 초기에 도메인을 제대로 이해하지 못했다면, 개발자가 혼자 끙끙대기보다 빠르게 프로토타입을 만들어 도메인 전문가와 함께 이터레이션을 돌면서 기준을 잡아가는 것이 100배 좋습니다.

재미있는 건, 프로젝트 초기에는 도메인 전문가조차 "어떤 질문과 답으로 이 Agent를 평가해야 할지" 모르는 경우가 많아요. 결국 개발자와 도메인 전문가가 함께 이터레이션을 돌며 기준을 잡아가는 과정이 필수적입니다.

사실 Agent 개발은 단순히 프롬프트 몇 줄 적는다고 뿅하고 나오지 않는 것 다들 알고 계실거에요. 에이전트 만들고 있는 수많은 엔지니어들 화이팅입니다!
