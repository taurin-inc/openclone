이번 프로젝트에서는 codex 를 많이 썼습니다. 클로드코드가 틀려서 codex가 해결한 일은 뭐였을까요? (감동한 나머지 codex 를 많이 써서 리밋에 걸려버렸습니다!) 프론트엔드 디자인과 전략적인 문제 해결이었습니다.

38가지나 되는 요구사항을 3일만에 개발해야 하는 일이 생겼습니다. 주저없이 linear mcp + 클로드코드로 티켓팅을 하고 클로드코드를 병렬 worktree를 생성하면서 작업했습니다.
그런데 이번에 만들어야 하는 서비스는 제가 특히 자신이 없었어요. "프로페셔널해보이는 UI 디자인"을 요구받았거든요. 공공기관에서 사용할 것이라 minimal 보다는 rich하고 뭔가 꽉차 보이는 디자인이 유리했어요. 

1. 똑같은 프롬프트로 UI 디자인을 맡겼더니 codex 압승
하지만 레퍼런스는 없었고, 저는 특히 프론트엔드에 약하다보니 이미 몇 번의 실패를 경험했습니다. 그래서 이번에는 프롬프트를 열심히 만들어서 클로드코드와 codex 둘 다 넣어봤어요. 결과는 codex의 압승. 클로드코드는 window97 같은 화면을 만들어줬는데 codex는 정말 마음에 쏙 드는 결과물이 나왔어요

버튼의 위치나 적절한 아이콘의 사용 등 "알아서 잘" 센스있게 해주더라구요. 오히려 codex가 코드 작성 후에 요약한 문장들에서 제가 배우고 있었어요. 덕분에 다음 프롬프트도 더 잘 쓸 수 있었습니다. 그렇게 선순환이!

2. "PDF 문서를 한 장씩 잘라서 각각 텍스트를 추출하고 이미지를 vision 인식하여 두 결과를 비교-종합해 결과를 도출하겠습니다"
제가 쓴 프롬프트가 아니라 codex가 스스로 접근한 방법입니다. 한 파일에 10장이 넘는 pdf 문서 2-30개에서 정확한 정보를 규격화해서 추출해야 했는데, 전략적인 접근을 고민하고 실행하라고 했더니 저렇게 하더라구요. 결과적으로 Claude 의 결과보다 훨씬 정확했습니다. 

3. codex는 크게크게 움직이는 느낌
한 쪽에는 claude code 다른 한 쪽에는 codex를 켜서 작업하면서 느낀 점이에요. 보폭이 크다라고 할까요? 클로드코드는 작업을 잘게 쪼개서 섬세하게 작업하는 느낌이라면, codex는 한 번에 큰 작업을 합니다. gpt-5와 opus 의 차이점이겠죠? 프롬프트의 차이일 수도 있고요. 하지만 gpt-5가 큰 작업을 할 수 있을만큼 자신감이 있는 모델이라는 건 알겠습니다. 위에서 말했던 것처럼 결과물도 좋고요.

완전히 codex로 갈아타지는 않을 겁니다. 아직까진 claude code가 툴을 더 잘 쓰고 실수가 적은 것 같아요. 생태계도 좋고요. 하지만 이제 클로드코드만 키지 않습니다. codex를 같이 켜서 서로 다른 종류의 작업을 시킵니다. 둘 중에 뭐든 생산성이 극대화되는 방향으로 사용해보려고요. 유저로서 너무 즐겁습니다!
