좋은 기회로 디캠프에서 바이브코딩을 주제로 얘기할 기회가 있었습니다. 30분이나 오셔서 놀랐는데요. 제가 완전히 잘못 짚었어요. 저는 사람들이 바이블 코딩을 할 이유를 찾지 못하고 있는 줄 알았거든요. 그런데 그게 아니라 하고 싶은데 막막한 거더라구요. 제가 준비한 자료가 포커싱을 잘 맞추지 못한 것 같아서 아쉬웠어요. 

또 개발자, 비개발자가 한 자리에 있었는데 비개발자분들한테 이야기를 맞추다보니 상대적으로 개발자분들이 소외된 것 같았어요. 끝나고나서 질문을 받았는데 너무 디테일해서 또 한 번 아차 싶었습니다. 시간 관계상 개발자분들에게 글로 이야기를 더 전달했습니다. 

오늘 즐거웠던 시간을 기록할 겸 또 다른 분들에게 도움이 될 수도 있으니 여기에도 남깁니다!

===

끝날 때 질문 개발자분들한테 질문을 받아보니, 개발 쪽으로 벨류를 못 드렸다는 생각이 들더라구요.

짧게 몇 가지를 적어보자면요

1. 바이브코딩은 “새로운” 프로젝트에서 훨씬 효과적이에요. 왜냐면 이미 사람이 짜놓은 코드베이스는 사람한테 맥락이 있는 경우가 많잖아요? 문서화도 생략했고, 주석도 안 달았고, 특히 제가 경험한 바로는 인프라가 치명적이었어요. 일부는 IaC, 일부는 GUI로 설정한 aws 리소스들이 있었는데 이걸 LLM에 이해시키는게 힘들었고, 그래서 전부 IaC로 바꿨습니다. 하지만 놀랍게도 전부 IaC로 바꾸는 고민을 한 시간보다 구현하는 시간이 더 짧았습니다..
2. 기존 코드베이스에 커서를 사용하는 건 조심할 게 하나도 없는데요. 클로드코드를 사용할 때는 하루치 토큰을 전부 문서화, 주석을 다는데 사용하시는 걸 추천드립니다. 또 그냥 코드만 보고 문서화를 시키는게 아니라 코드에는 안 담겨있는 인간이 알고 있는 비즈니스로직도 다 알려주세요.
3. 기존 코드베이스에 바이브코딩을 할 때는 의존성이 최대한 적은 것들부터 시작하세요. 그걸 이 영상에서는 Leaf nodes 라고 부르는데 영상이 너무 설명을 잘해줘서 [영상을 추천드립니다!](https://lnkd.in/gCtdWAxv) 코어 아키텍처에 적용하는 코드를 내가 검토하지 않고 적용하는 건 너무 위험합니다! 
4. 개발자분들 중에 커서나 copilot를 이미 사용하고 있다면 클로드코드를 사용하는데 어려움이 없을 거에요. 근데 꼭 꼭 써보셔야돼요. 툴 콜링하는 능력이 미쳤습니다. 괜히 LLM provider들이 CLI만 만드는게 아니에요. 이게 미래..
5. CC를 써보셨다면 부차적인 팁은 큰 의미가 없는 것 같아요 사실. mcp 를 여러 개 연결해라, super claude를 써라, subagents 이런 것들이요. 이런 건 CC의 효과를 느끼면 알아서 찾아보고 적용하시더라구요.
6. 오히려 저는 CC를 사용하면서 Deep한 나만의 개발 환경을 만드는게 더 중요하다고 생각합니다. git worktree 를 저는 CC를 사용하기 전에는 알지도 못했어요. CC를 병렬로 쓰려고 LLM이랑 대화하다보니까 알게 됐어요. 개발 블로그 하시는 아웃사이더님도 최근에야 써보셨다고 할 정도인데, 이런 딥한 기술들이 생산성을 높이는데 정말 도움이 돼요. 
    1. 저는 linear issue 이름으로 worktree를 생성하면 자동으로 linear issue 내용을 불러와서 task.md 파일을 생성하고 이 파일을 분석하면서 세션을 시작하는 워크플로우를 만들었어요. 병렬로 여러 작업을 돌려버리기 너무 편합니다. 이제는 밤에 cron 을 돌려볼까 궁리도 해요.. 구독료가 아까우니까!
7. 클로드코드가 한 번 돌아가면 턴(대화의 수)이 엄청 많거든요. 그래서 클로드코드에서 대화를 많이 하는 것보다는 미리 요청을 구조화해서 크게 던지는 게 더 효율적이라고 느꼈습니다! 잘 구조화해서 던지면 planning mode 보다 더 효과적이에요

오늘 발표 들어주셔서 감사합니다!
