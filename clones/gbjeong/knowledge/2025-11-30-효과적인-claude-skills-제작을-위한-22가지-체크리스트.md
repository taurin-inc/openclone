Claude Skills는 MCP와 달리 직접 만들어야 효과적인데요. 그래서 잘 만들었는지 직접 평가해야만 하는데요. Anthropic 에서 "Checklist for effective Skills"을 올려주었는데 공식 한글 번역이 안 좋아서 다시 번역해봤습니다. 22가지를 체크리스트를 확인해보세요.

[핵심 품질 10가지]
1. description이 구체적이며 핵심 키워드를 포함한다
- 설명: 메타데이터의 description 필드에 "문서 작업 도움" 같은 모호한 표현 대신, Skill의 기능과 활성화 맥락을 구체적으로 명시해야 합니다.

2. description에 "무엇을 하는지"와 "언제 사용하는지"가 모두 명시되어 있다
- 설명: description은 두 가지 질문에 답해야 합니다: "이 Skill이 제공하는 기능은?" 그리고 "Claude가 언제 이것을 선택해야 하는가?"

3. SKILL.md 본문이 500줄 미만이다
- 설명: 기본 지침 파일은 간결해야 합니다. 대화 기록과 다른 정보를 위한 컨텍스트 윈도우 효율성을 유지하기 위함입니다.

4. 상세 내용은 별도 파일로 분리되어 있다 (필요시)
- 설명: 내용이 너무 길어지면, 보조 정보를 별도 파일로 분리하고 점진적 공개 패턴을 사용해 참조합니다.

5. 시간에 따라 변하는 정보가 없다 (있다면 "이전 패턴" 섹션에 배치)
- 설명: 날짜나 기간이 포함된 지침은 시간이 지나면 틀린 정보가 됩니다. 레거시 방식은 접힌 "old patterns" 섹션에 배치합니다.

6. 문서 전체에서 용어가 일관되게 사용된다
- 설명: 하나의 개념에 하나의 용어를 선택하고 일관되게 사용합니다. 동의어를 혼용하면 다른 개념인지 혼란을 줍니다.

7. 예시가 추상적이지 않고 구체적이다
- 설명: 예상 동작에 대한 이론적 설명 대신, 실제 입출력 쌍과 현실적인 시나리오를 제공합니다.

8. 파일 참조 깊이가 1단계를 넘지 않는다
- 설명: 참조되는 모든 파일은 SKILL.md에서 직접 연결되어야 합니다. 중첩 참조(파일→파일→파일)는 Claude가 부분 읽기만 하게 될 위험이 있습니다.

9. 점진적 공개(Progressive Disclosure) 패턴을 적절히 활용한다
- 설명: Claude가 필요한 정보만 로드하도록 구조화합니다. SKILL.md는 상세 리소스로 안내하는 진입점 역할을 합니다.

10. 워크플로우가 명확한 단계로 구성되어 있다
- 설명: 복잡한 절차는 Claude가 작업 실행 중 추적할 수 있는 번호가 매겨진 순서나 체크리스트로 작성합니다.

[코드 및 스크립트 8가지]
1. 스크립트가 문제를 Claude에게 떠넘기지 않고 직접 해결한다
- 설명: 예측 가능한 오류 상황은 코드 내에서 명시적으로 처리합니다. Claude가 해결책을 생성하도록 떠넘기지 않습니다.

2. 오류 처리가 명시적이며 해결 방법을 안내한다
- 설명: 문제 발생 시 스크립트가 구체적이고 조치 가능한 오류 메시지를 제공하여 해결 방법을 안내합니다.

3. 매직 넘버 없이 모든 상수값에 근거가 명시되어 있다
- 설명: 하드코딩된 모든 숫자나 설정값에는 근거가 문서화되어야 합니다. 설명할 수 없는 값은 Claude도 판단할 수 없습니다.

4. 필수 패키지가 명시되고 실행 환경에서 사용 가능함이 확인되었다
- 설명: 의존성을 명시적으로 선언하고, 지원되는 코드 실행 환경에서 사용 가능한지 확인합니다.

5. 스크립트에 목적, 파라미터, 출력 형식이 문서화되어 있다
- 설명: 목적, 파라미터, 예상 출력 형식을 설명하는 주석을 포함합니다.

6. 경로에 백슬래시(\) 대신 슬래시(/)를 사용한다
- 설명: Unix 스타일 경로 표기법을 일관되게 사용합니다. Windows 백슬래시는 Unix 기반 시스템에서 실패합니다.

7. 중요한 작업에 검증/확인 단계가 포함되어 있다
- 설명: 고위험 작업(일괄 업데이트, 데이터 수정)은 최종 실행 전 중간 검증 단계를 포함해야 합니다.

8. 품질이 중요한 작업에 피드백 루프가 구축되어 있다
- 설명: 검증 → 오류 탐지 → 수정 사이클을 구축하여 Claude가 출력을 반복적으로 개선할 수 있게 합니다.

[테스트 4가지]
1. 최소 3개의 평가 시나리오가 작성되었다
- 설명: 문서화를 완료하기 전에 Skill이 실제 문제를 해결하는지 입증하는 테스트 시나리오를 개발합니다.

2. Haiku, Sonnet, Opus 모델에서 각각 테스트되었다
- 설명: 모델 티어별로 기능을 검증합니다. Haiku는 Opus보다 더 명시적인 가이드가 필요합니다.

3. 실제 사용 시나리오로 테스트되었다
- 설명: 인위적인 테스트 케이스가 아닌 실제 작업으로 평가합니다.

4. 팀의 피드백을 반영했다
- 설명: 의도된 사용자와 Skill을 공유하고, 명확성과 완성도에 대한 인사이트를 반영합니다.
