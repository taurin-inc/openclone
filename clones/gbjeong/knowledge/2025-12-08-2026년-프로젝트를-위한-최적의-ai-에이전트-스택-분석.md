2026년에는 어떤 Agent Stack이 내 프로젝트에 맞을까?

대규모 Agent 파이프라인을 만들 때:
Google ADK, 단점 TS가 없다. 대신 Python, Go, Java를 지원한다. 누구를 타겟팅했는지 여지없이 보여주는 언어 선택이다. 

유저가 직접 세밀하게 조율할 수 있는 Agent를 빨리 만들 때: 
Claude Agent SDK. 성능이 좋은데 이 SDK보다 더 빨리 에이전트를 구현할 수 있는 건 없는 것 같다. 단점은 Claude 만 사용할 수 있다는 것 

모델: 다 써보고 eval 해서 좋은 것 쓰면 된다

LLM Proxy: LiteLLM

Prompt versiong: git. 저는 프롬프트의 변경과 노드/routing 등 코드의 변경이 같이 이루어지는 경우가 많아서 프롬프트만 따로 관리하게 되면 오히려 추적이 어려웠습니다.

LLM Tracing & Eval Framework: 
langfuse. opensource라는게 매우 큰 장점. Agent 를 많이 호출하면 trace 수로 과금이 늘어나는 것도 문제인데 생각보다 Tracing 서비스가 배포된 클라우드로 outbound traffic 이 많이 발생된다. 

처음엔 도대체 어디서 트래픽이 빠져나가는지 몰라서 당황했는데, 특히 langsmith 는 말도 안되게 큰 outbound 트래픽을 안겨줬다. 
그래서 오픈소스로 내가 사용하는 클라우드 안에서 사용할 수 있다는게 부담이 많이 줄어든다. 오픈소스라는 측면에서는 Arize에서 만든 Phoenix도 고려할 만하다. 

Repository 관리 전략: Monorepo. fe+be 같이 소극적인 monorepo가 있는 반면, 거의 모든 Repository 를 하나에 담아버리는 경우도 있다. 전자는 매우 선호하는 편이고, 후자는 지금 시작하는 프로젝트에서 훌륭한 엔지니어분들과 도전해볼 예정이다.

Language: whatever!
- 1년 전까지만 해도 python 이라고 말했을텐데, 무슨 언어를 사용해도 상관 없는 것 같아요. 오히려 Korean 이면 되는데, 무슨 언어를 고르고 있냐는 농담도 합니다.
- 하지만 위에 언급한 생태계를 잘 담으려면 Python이 일반적으로 좋은 선택지이다. 물론 Agent Framework 는 상대적으로 레이어가 작아서 익숙한 TS 를 고르고 TS를 지원하는 프레임워크를 선택하거나 아예 프레임워크 없이 구현하는 것도 충분히 가능하다. 클로드코드가 더 잘해줄 수도 있다.

사실 최고의 스택은 없다는 것 다들 알고 계실 겁니다. 서비스마다, 기존 코드베이스 때문에, 조직이라는 환경 때문에 더 좋은 스택은 늘 변하니까요. 그럼에도 불구하고 좋은 스택을 선택하기 위해서 엔지니어들은 정말 치열하게 고민합니다. 

+저는 프로덕션 환경에서 langgraph를 운영해봤지만 저에게는 맞지 않았습니다. 프레임워크의 틀을 벗어나서 pure한 코드를 쓰는 케이스가 늘어나더라구요. 제가 잘 사용하지 못해서 그런 것이겠지만 LangGraph 로 구현된 에이전트를 여러가지 용도를 위해 서빙하는 프레임워크에 얹으면 구조가 점점 이상해지더라구요. 이건 프레임워크라는 것 자체의 문제인 것 같긴한데, LangGraph 가 너무 쉬운 프레임워크라서 역설적으로 내가 푸는 문제에 핏하게 만드는 것에 답답함을 느꼈던 것 같습니다.

이 글이 스택을 선택하는 엔지니어 분들한테 작은 도움이 되길 바랍니다!
