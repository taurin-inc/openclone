PM + Claude Code = 무적의 프로토타이핑. 이번에 우리 플랫폼에서 한 PM을 보며, “AI 시대에 PM의 모든 컨텍스트를 주입한 프로토타입을 만드는 것이다”는 확신을 얻었다. 이 분은 자료 정리 조차 직접 분석하는 대신, o3, Opus, NotebookLM, Claude Code를 종횡무진하며 프로토타입을 만들었다. 무엇보다 인상 깊었던 건 LLM 모델이 효과적으로 동작하도록 자료를 AI-native하게 재구성하고 컨텍스트를 치밀하게 주입하는 방식이었다. 이 접근은 마이크로소프트 CPO가 강조하는 원칙—“AI로 프로토타이핑하지 않는다면 잘못하고 있는 것이다”, “프롬프트 세트가 새로운 PRD다”, “메모보다 데모”—와 정확히 맞닿아 있다. 그녀는 아이디어를 가장 빠르고 정확히 전달하려면 작동하는 데모와 프롬프트 세트로 핵심 상호작용을 정의하라고 말한다.

PM이 맡은 제품은 생판 모르는 화학 전문 도메인이었다. 하필 클라이언트에게 받은 자료는 기존 시스템의 동영상 자료였는데, 모든 동영상을 일괄 mp3로 변환해 NotebookLM에 올리고 1차 STT를 돌린 뒤, 도메인 용어집과 v2 이미지 레퍼런스를 컨텍스트로 주입해 잘못 인식된 부분을 반복 교정했다. 이렇게 만든 스크립트 묶음이 곧 제품의 스토리보드가 되었고, 팀 공유용 가이드의 초석이 됐다. 포인트는 “자료를 모델이 이해하기 쉬운 형태로 먼저 바꿔라”였다. 텍스트화→컨텍스트 주입→오류 교정 사이클을 자동화한 덕분에, 이후 모든 산출물이 같은 용어와 같은 의미로 정렬되기 시작했다.

이미지 자료도 같은 흐름이었다. 흩어진 동영상 스크린샷을 순서대로 하나의 PDF(v1-guide.pdf)로 묶어 o3와 Opus에 각각 투입하고 Markdown으로 변환했다. 결과물을 비교해 보니 o3가 문맥 보존과 서술 흐름에서 더 안정적이어서, Opus와 GPT-4o의 산출물을 다시 o3에 넣어 교차 검증시키고 최종 가이드를 출력했다. 이렇게 얻은 정제 결과를 v1-guide-parsed.md로 작성했고, 여기서 프롬프트 세트(핵심 상호작용, 제약, 예외 규칙, 실패 사례)를 추출해 이후 논의의 기준선으로 삼았다. 복잡한 설명보다 “데모 + 프롬프트 세트”로 대화하니 합의 속도가 눈에 띄게 빨라졌다. 체나프라가다도 “프롬프트 세트가 새로운 PRD이며, 메모보다 데모를 가져오라”는 메시지를 전했다.

다음은 Claude Code로의 오케스트레이션이다. 스택은 Vercel, MUI, PostgreSQL, Neon MCP, Playwright MCP(OCR-AI는 초기엔 mock 데이터). 수동으로 한 일은 Neon MCP 세팅과 Vercel CLI 인증뿐이었다. 상위 명령은 단순했다. “v2 PRD를 구현하라. 사용자 시나리오는 v1 가이드를 따른다.” 그러자 기본 화면과 주요 플로우가 빠르게 올라왔다. 레이아웃이 들쑥날쑥하자 Playwright MCP로 레퍼런스 서비스의 스타일을 관찰해 규칙을 요약하게 하고, 그 규칙에 맞춰 컴포넌트를 재정렬하라고 지시했다. 파일 업로드 등 일부 컴포넌트가 동작하지 않는 버그는 원인 후보를 나열→수정 패치 생성→적용→검증 루프를 돌렸다. 
이렇게 자료 → 구조화 → 프롬프트 세트 정교화 → PRD → 구현 → 배포 → 테스트로 이어지는 전 과정을 데모 중심으로 재배치하니, PM은 프로세스를 ‘관리’하기보다는 무엇을 밀고 무엇을 접을지 고르는 편집과 Taste 역할에 시간을 쓸 수 있었다. 마치 Sam Altman 이 말하는 Idea Guy 같았다.

정리. 
이 PM이 보여준 것은 직접 하는 일보다 쉬운 일이다. 따라하기만 하면 생산성이 엄청나게 올라간다.
1. 자료를 각종 AI를 사용해 변환 및 리서치하고
2. 컨텍스트 번들을 꾸준히 주입하며
3. “데모 + 프롬프트 세트”로 팀의 언어를 맞추고
4. Claude Code에 상위 지시를 내린 뒤
5. 사람은 위 과정을 반복하며 편집과 선택에 집중하는 것이다.
