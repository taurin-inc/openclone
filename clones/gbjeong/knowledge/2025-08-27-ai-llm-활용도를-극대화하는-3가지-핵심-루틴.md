AI를 잘 쓰고 있는지 점검하는 저만의 루틴이 있습니다. 일명 LLM 뽕 뽑는 방법!

1. LLM이 읽을 것이 많은지 점검한다 +MCP
노션, 옵시디언, github 등 작업할 때 필요한 정보를 LLM이 접근할 수 있는지 확인합니다. 작업을 할 때 접근할 mcp 를 세팅하고 각 mcp 툴을 어떤 로직으로 호출해야 하는지 미리 지정해둡니다. 

CLAUDE.md 를 사용하면 편리해서 코딩 작업 이외에도 클로드코드를 많이 씁니다. 아니면 클로드의 프로젝트 기능이나 chatgpt의 프로젝트 기능을 씁니다. 지침을 디테일하게 적어놓고 그에 맞춰서 요청을 보낼 수 있으니까요. 하지만 요새는 mcp 때문에 클로드를 많이 쓰게 됩니다.

2. 명확한 룰 세팅
데이터를 확인했다면 내가 시키는 태스크의 룰이 명확하게 설계합니다. 목표, 판단 기준, 루브릭까지 주로 사용하는 태스크마다 분리된 규칙이 있어야 잘 동작합니다.
좋은 규칙을 설정했다면 결과물이 재현 가능해야 합니다. 우리가 프롬프트/컨텍스트를 “엔지니어링”이라고 부르는 이유는 과학적으로 설계하여 재현 가능하게 만들기 때문입니다. 에이전트를 슬롯머신처럼 사용해서는 안 됩니다.

번외로, OpenAI SWE Benchmark 데이터셋을 만들 때 사용한 instruction guide를 참고하기도 합니다. 이는 LLM의 코딩 능력을 측정하는 벤치마크 데이터셋을 구성할 때 인간에게 제공된 가이드 문서인데, 구조가 체계적으로 되어 있습니다.

3.	LLM이 쓸 것이 많아야 함
코드, 글, 이미지, 영상 등의 결과물을 만들고 있다면 LLM을 활용할 수 있습니다. LLM이 초안을 작성하면 토큰 사용량이 확실히 늘어납니다.
이미지나 영상을 제작할 때 기본 UI는 한 번에 여러 버전을 만들어 채택하는 것입니다. 마찬가지로 글을 쓸 때에도 여러 버전을 생성하게 설정하고, 그중 가장 나은 것을 선택하여 발전시키는 방식이 효과적입니다.

또한 같은 콘텐츠를 멀티 플랫폼으로 복제할 수도 있습니다. 여러 SNS에 동시에 올릴 글을 생성하는 것은 물론이고, 예를 들어 코드를 작성해 소프트웨어를 만들었다면 내부에 공유할 슬랙 메시지도 AI가 작성할 수 있습니다. 이 기능을 사용자에게 알리는 릴리즈 노트까지 작성할 수 있습니다.
