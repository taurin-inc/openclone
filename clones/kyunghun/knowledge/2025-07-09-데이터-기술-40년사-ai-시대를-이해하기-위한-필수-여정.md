데이터 기술 40년사: AI 시대를 이해하기 위한 필수 여정

요즘 AI를 공부하다보니 데이터베이스의 중요성을 느끼게 되었고, SQL부터 시작해서 잘 모르는 용어들, 그리고 스노우플레이크나 데이터브릭스 같은 회사들이 얼마나 대단한 곳인지 핵심을 이해하지 못하는 것 같아서 전체 흐름을 정리해보았습니다.

전문가 분들에게는 평이한 내용이지만, 이 섹터에 관심있으신 분들께 도움이 될까 싶어 공유드립니다.

1. 1980~90년대: 정확한 거래의 시작

은행 입출금이나 재고관리에서 가장 중요한 것은 '정확성'이었습니다. 1만원이 들어왔으면 정확히 1만원이 기록되어야 했습니다.

Oracle, IBM DB2 같은 관계형 데이터베이스가 등장했습니다. 엑셀 표처럼 행과 열로 데이터를 정리하되, 훨씬 더 엄격한 규칙을 적용했습니다. SQL과 테이블 기반 구조는 관계형 데이터베이스의 핵심이 되었고, 지금도 대부분의 데이터베이스가 이 방식을 따릅니다.

*관계형 데이터베이스: 데이터를 행과 열로 구성된 테이블 형태로 저장하고, 각 테이블 간 관계를 공통된 키로 연결하는 구조. 정형화된 스키마 기반으로 데이터 정확성을 유지하며, SQL로 검색과 연산 가능
*스키마: 데이터베이스 내 테이블 구조 설계도 (각 컬럼 이름, 타입 등)
*SQL: 관계형 데이터베이스 정보를 조회, 삽입, 수정, 삭제하기 위한 표준 언어

2. 2000년대: 빅데이터의 탄생

구글, 야후 같은 초기 인터넷 기업들이 예상치 못한 문제에 부딪혔습니다. 웹사이트 방문자가 급증하면서 기존 데이터베이스로는 동시 접속자를 감당할 수 없게 된 것입니다. 수평 확장과 캐싱 기술이 발전했지만 임시방편에 불과했습니다. 페이스북 같은 SNS도 등장하면서 하루에 수천만 건의 게시글, 좋아요가 쌓였고, 기존 데이터베이스로는 감당할 수 없었습니다.

이때 두 가지 혁신이 일어났습니다.

첫째, NoSQL 데이터베이스(비관계형)가 등장했습니다. 기존의 엄격한 표 구조를 포기하고, 다양한 형태로 데이터를 저장할 수 있게 되었습니다. 마치 정형화된 양식 대신 자유로운 메모장을 쓰는 것입니다.

둘째, 빅데이터 처리 기술(Hadoop)이 등장했습니다. 분산 컴퓨팅으로 여러 대의 저렴한 서버를 동시 활용하고, 데이터를 작은 블록으로 나누어 병렬 처리하는 구조를 구현했습니다. 수십~수백 대의 컴퓨터를 하나의 거대한 컴퓨터처럼 작동시켜 기존에 불가능했던 규모의 데이터를 빠르고 저렴하게 처리할 수 있게 되었습니다.

하지만 트레이드오프가 있었습니다. NoSQL은 속도와 확장성을 얻는 대신 여러 데이터를 연결해서 분석하는 것이 복잡해졌습니다. Hadoop은 대용량 데이터를 잘 저장하지만, 분석하려면 복잡한 프로그래밍이 필요했습니다. 비즈니스팀에서는 "SQL로 간단히 분석하고 싶다"고 불만을 토로했습니다.

3. 2010년대 초반: 분석 전용 클라우드 서비스

빅데이터를 SQL로 쉽게 분석할 수 있는 클라우드 서비스들이 등장했습니다. 구글 BigQuery와 AWS Redshift가 대표적입니다. 이들은 클라우드 데이터 웨어하우스라고 불리며, 대용량 데이터를 SQL로 빠르게 분석할 수 있게 해주었습니다. 예를 들어, 전국 매장의 6개월치 매출 데이터에서 "서울 지역 30대 고객의 구매 패턴"을 몇 분 안에 뽑아낼 수 있습니다.

하지만 미리 정해진 크기의 서버를 항상 대기 상태로 유지해야 해서, 분석 작업을 하지 않는 시간에도 비용이 발생했습니다. CFO들은 쓴 만큼만 내고 싶다고 불만을 터뜨렸습니다.

4. 2010년대 후반: 데이터 민주화

이때 스노우플레이크와 데이터브릭스가 등장합니다.

스노우플레이크는 스토리지와 컴퓨팅을 분리하는 혁신적 구조를 제시했습니다. 데이터는 안전한 창고에 보관해두고, 분석이 필요할 때만 컴퓨팅 자원을 켜서 처리하는 방식입니다.

데이터브릭스는 데이터 레이크하우스 개념을 제시했습니다. 데이터 레이크와 데이터 웨어하우스의 장점을 결합한 개념으로, 숫자와 텍스트뿐만 아니라 이미지, 동영상, 음성 파일, 로그 파일까지 형태를 가리지 않고 모두 저장한 뒤, Apache Spark로 필요에 따라 분석하는 방식입니다.

결과적으로 분석 영역(OLAP)에서 획기적인 혁신을 이루어냈습니다. 기업들은 사용한 만큼만 비용을 지불하면서도, 기존보다 훨씬 빠르고 쉽게 대용량 데이터를 분석할 수 있게 되었습니다. 비즈니스 분석가들이 복잡한 기술 지식 없이도 직접 데이터를 탐색할 수 있는 "데이터 민주화"가 시작되었습니다.

*OLAP: 다량의 데이터 기반 비즈니스 분석을 위한 조회 중심 시스템
*OLTP: 은행 이체, 주문 처리 같은 실시간 거래 처리 시스템

5. 2020년대: 서버리스화 확대

실시간 거래 처리용 데이터베이스(OLTP)마저 "스노우플레이크처럼 편하게 쓰고 싶다"는 요구가 나왔습니다. 서버리스 운영 데이터베이스들이 등장하기 시작했습니다.
•PlanetScale: MySQL 기반으로, 트래픽이 없을 땐 리소스를 자동으로 줄이고, 스키마 변경을 Git처럼 안전하게 관리할 수 있는 기능 제공
•Neon: PostgreSQL 기반으로, 스토리지와 컴퓨팅을 분리하고 데이터베이스 브랜치를 초고속으로 생성할 수 있는 아키텍처 제공

*MySQL: 웹 서비스에서 널리 사용, 속도와 배포 용이성을 중시해 빠르게 구축하기 적합 *PostgreSQL: SQL 표준 준수와 데이터 무결성을 최우선, 복잡한 쿼리 및 확장 기능이 풍부하지만 무거움

또한 OLTP와 OLAP를 하나의 시스템에서 동시에 처리하는 HTAP도 주목받고 있습니다. 이는 지난 40년간 분리되어 있던 "실시간 저장용"과 "분석용" 데이터베이스를 통합하려는 시도입니다. 거래가 발생하는 즉시 분석이 가능해져, 실시간 의사결정과 운영 복잡성 해결에 큰 잠재력을 보여주고 있습니다.

6. 변하지 않는 본질

지난 40년간 데이터 기술 발전을 돌아보면, 결국 데이터를 쉽게 저장하여 다량의 데이터를 넣는 것과 분석을 효율적으로 저렴한 비용으로 빠르게 하는 것, 이 두 가지 방향으로 일관되게 발전해왔습니다.

AI 시대가 본격화되면서 이런 요구는 더욱 중요해지고, 혁신은 더욱 가속화될 것입니다.
