팔란티어는 어떻게 '맥락'을 데이터로 저장할까?

온톨로지라는 단어가 어딘가 추상적이고 거창하게만 느껴졌는데, 실제로는 분명한 데이터 구조가 있을 것 같다는 생각에 찾아보게 되었습니다. 

=== 핵심 요약 ===

1. 다른 데이터베이스처럼 컬럼과 로우 중심으로 구성되어 있는 물리 계층이 존재한다.
2. 물리계층은 일반적 객체 테이블에 더하여 관계 테이블, 계보 테이블이 존재하고, 버져닝과 파이프라인을 통해 시간과 맥락을 기록한다.
3. 물리계층 위에 온톨로지라는 계층을 한층 더 쌓았고, 자연어/비지니스 언어와 물리 계층의 각종 데이터 사이에 관계를 메타 데이터로 구축하였으며, 자연어로 물어보면 물리계층의 데이터를 찾을 수 있는 쿼리 컴파일러 같은 것을 구현했다.
4. 이를 통해 비지니스의 맥락이 포함된 디지털 트윈을 만들 수 있었고, 비지니스 사이드에서도 SQL등의 문법을 몰라도 필요한 데이터를 아주 쉽게 찾을 수 있다.

실제 시스템은 제가 이해할 수 있는 범위를 훨씬 넘어서겠지만, 핵심적으로 보이는 부분만 최대한 간추려보았습니다.

=== 상세 내용 ===

0. 팔란티어의 데이터 구조
우선 크게 두 개의 계층으로 구성되는 것 같습니다. 실제 데이터를 저장하는 물리 계층과 이를 비즈니스 관점에서 활용할 수 있게 해주는 온톨로지 계층입니다.

1. 물리 계층
물리 계층은 실제 데이터가 저장되고 처리되는 레벨로, 컬럼과 로우 중심의 전통적인 데이터베이스 구조입니다. SQL 쿼리, 버저닝, 관계 테이블, 파이프라인 실행 결과 등이 이 계층에서 직접 관리되며, 성능과 신뢰성을 담당합니다.

물리 계층을 이해하는데 핵심적인 요소는 3개의 테이블, 그리고 2개의 개념이 있는 것 같습니다.

1) 객체 테이블
하나의 테이블에 '부품', '직원' 등 실제 데이터를 행(Row) 단위로 저장하며, 열(Column)은 속성을 나타냅니다. 각 객체는 고유한 ID를 가지며, 이 ID를 통해 다른 테이블들과 연결됩니다.
・ Column: 부품ID, 부품명, 제조일, 상태, 공급업체, 배치번호
・ Row: 각 부품별 정보가 한 줄씩 저장 (A1234 볼트, 2024-05-01, 정상, 삼성부품, B9999)

2) 관계 테이블
객체들 간의 관계를 저장하는 테이블로, '부품 A를 직원 B가 검사함' 같은 연결 정보와 그 맥락을 함께 저장합니다. 이것이 팔란티어가 컨텍스트를 캡처하는 핵심 방식입니다.
・ Column: 부품ID, 직원ID, 관계유형, 역할, 검사일시, 유효시작일
・ Row: 각 관계별 정보가 한 줄씩 저장 (A1234 볼트, 김철수, 검사함, 품질검사원, 2024-06-01 10:00, 2024-06-01)

3) 버저닝 (Versioning)
데이터가 바뀔 때마다 이전 값도 함께 저장하여, 언제 어떤 값이 있었는지 시간별로 추적할 수 있도록 하는 기능입니다. 이는 "언제 이 문제가 시작되었는지" 같은 시간적 컨텍스트를 제공합니다. 버저닝은 위 두개 객체/관계 테이블에 컬럼값으로 추가됩니다.

4) 계보 테이블 (Lineage Metadata Table)
어떤 데이터셋이 어떤 파이프라인에서 만들어졌고, 어떤 데이터셋을 입력으로 사용했는지를 기록하여 데이터 흐름을 추적할 수 있게 해주는 테이블입니다. 이를 통해 "이 결과가 어떻게 나왔는지" 맥락을 제공합니다.
・ Column: 변환ID, 입력데이터셋, 출력데이터셋, 실행일시, 실행자, 상태
・ Row: 각 파이프라인 실행마다 한 줄씩 저장

5) 파이프라인
원시 데이터를 정제하거나 분석하는 자동화된 처리 흐름으로, 실행할 때마다 새로운 데이터 버전을 생성하고 계보를 남깁니다. 비즈니스 로직이 어떻게 적용되었는지의 컨텍스트를 제공합니다. 예를들면 Transform T1234 라는 파이프라인은 raw_parts에서 온도 범위 검증과 결측값 처리를 거쳐 cleaned_parts를 만듭니다. 

2. 온톨로지 계층
온톨로지 계층은 사용자가 데이터의 구조를 비즈니스 개념 중심으로 이해하고 탐색할 수 있도록 돕는 의미 기반 추상화 계층입니다. 객체와 관계를 메타데이터로 정의하고, 쿼리 컴파일러를 통해 자연어 또는 의미 중심 질의를 SQL로 자동 변환해주는 역할을 합니다.

1) 메타데이터 (객체/관계 정의 포함)
각 객체가 어떤 속성들을 갖고 어떤 관계를 맺는지를 사람이 이해하기 쉽게 구조화해 YAML이나 GUI로 정의한 정보입니다. 비즈니스 담당자가 기술적 세부사항 없이도 데이터를 이해할 수 있게 해주는 핵심 요소입니다.
예를들면 Part 객체는 "부품"이라는 이름으로 정의되고, serial_no는 "일련번호", status_code는 "상태"로 표현되어 비즈니스 담당자도 쉽게 이해할 수 있습니다. 핵심은 기술적인 테이블 구조를 비즈니스 언어로 추상화하여 누구나 이해할 수 있게 만든다는 점입니다.

2) 쿼리 컴파일러 (자연어/의미 기반 쿼리 처리)
사용자가 "불량 부품과 검사자 목록 보여줘"처럼 의미 중심으로 질의하면, 이를 내부적으로 복잡한 SQL JOIN으로 변환해 물리 데이터에서 실행하게 해주는 자동 번역기입니다. 이것이 비즈니스 담당자도 쉽게 사용할 수 있게 해주는 핵심 기술입니다.
예를들면 "불량 부품을 검사한 직원들의 이름"이라는 질의를 parts_objects, part_employee_edges, employees_objects 세 테이블을 JOIN하는 SQL로 자동 변환합니다.

그 외에도 복잡한 기술들이 많지만, 결국 핵심은 컨텍스트 데이터 수집이라는 철학을 중심으로 설계되어 있는 것 같습니다. 

객체 테이블은 개별 사실을 저장하고, 관계 테이블은 사실들 간의 연결과 맥락을 기록하며 버저닝은 시간적 맥락을, 파이프라인은 비즈니스 로직 적용 맥락을 보존하고, 계보 테이블은 데이터 생성 맥락을 추적합니다. 그리고 온톨로지 계층의 메타데이터와 쿼리 컴파일러가 이 모든 컨텍스트를 비즈니스 담당자도 쉽게 활용할 수 있게 해줍니다.
