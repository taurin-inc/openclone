AI 에이전트가 PLG를 어떻게 무너뜨리고 있는가

1. 사용자가 제품을 '사용'하지 않고 '위임'하기 시작하면서, 기존 PLG 전략들이 재정의되어야 할 시점에 왔다고 합니다.

2. 전통적인 PLG는 사용자가 직접 제품을 사용하며 가치를 발견하는 여정에 기반했습니다. 예를 들어 Notion을 처음 사용한다면 가입 후 빈 페이지에서 시작해서 텍스트를 입력해보고, 블록을 추가해보고, 템플릿을 둘러보고, 데이터베이스를 만들어보면서 점점 "이거 정말 유용하네!"라고 느끼게 됩니다.

3. 이 과정에서 기업들은 온보딩 플로우, UI 최적화, 점진적 기능 발견을 통해 사용자를 활성화시키는 전략을 구축했습니다. "사용자가 10번 클릭했는가?", "5개 기능을 써봤는가?" 같은 지표로 성장을 측정했습니다.

4. 하지만 AI 에이전트 시대에는 게임이 바뀝니다. 이제 사용자는 '실행자'에서 '위임자'로 역할이 바뀝니다. "프로젝트 관리 시스템 만들어줘"라고 말하면 에이전트가 알아서 페이지 구조, 데이터베이스, 템플릿을 다 만들어줍니다. 사용자는 클릭도 안 하고 바로 완성된 결과물을 받게 되죠.

5. 이러한 변화는 기존 PLG 지표들을 무력화시킵니다.

첫째, 온보딩 플로우가 의미를 잃습니다. "여기가 메뉴고, 여기서 파일 업로드하고..." 같은 인터랙티브 투어나 툴팁을 AI 에이전트는 필요로 하지 않습니다.

둘째, UI 최적화의 임팩트가 줄어듭니다. 버튼 위치를 조정하고 클릭 수를 줄이는 데 공들였지만, 에이전트는 API를 통해 작업하거나 UI를 완전히 우회할 수 있습니다.

셋째, 기능 발견이 줄어듭니다. 기존에는 사용자가 메뉴를 돌아다니며 "어? 이런 기능도 있네?"라고 우연히 발견하는 경우가 많았습니다. 하지만 에이전트는 요청받은 작업만 정확히 수행하므로 사용자가 다른 기능을 발견할 기회가 사라집니다.

넷째, "아하 모멘트"와 사용량 기반 활성화 지표가 약해집니다. 기존에는 "사용자가 파일 10개를 업로드했을 때" 같은 점진적 사용 패턴을 추적했지만, 에이전트는 한 번의 위임으로 모든 가치를 즉시 전달할 수 있습니다.

6. 그렇다면 어떻게 대응해야 할까요? Wing VC에서는 네 가지 구체적인 방향을 제시합니다.

첫째, 기능 참여보다는 의도 포착에 최적화하기. 기존의 "10번 클릭했나? 5개 기능 써봤나?" 같은 지표 대신, 사용자가 원하는 것을 명확히 표현할 수 있게 도와주는 데 집중해야 합니다. ChatGPT처럼 "구체적으로 어떤 보고서가 필요하세요?"라고 묻거나, 드롭다운으로 목적을 선택하게 하는 방식입니다.

둘째, 결과물을 측정 가능하고 공유 가능하게 만들기. 클릭수나 체류시간 같은 과정 지표 대신 실제 만들어진 결과물을 추적해야 합니다. "AI가 작성한 이메일 3개", "생성된 보고서 5개" 이런 식으로 성과를 측정하고, 이를 팀원들에게 쉽게 공유할 수 있게 만드는 것입니다.

셋째, 온보딩을 '위임 설정'으로 재구성하기. 기존의 "여기가 메뉴고, 여기서 파일 업로드하고..." 같은 UI 설명 대신, AI가 일을 잘하도록 환경을 세팅하는 데 집중해야 합니다. "구글 캘린더 연결하기", "회사 데이터 업로드하기", "선호하는 보고서 스타일 설정하기" 같은 방식입니다.

넷째, 신뢰와 투명성, 되돌리기 가능성을 설계하기. 사용자가 AI에게 위임하는 것에 대한 불안감을 해소해야 합니다. AI가 어떤 단계로 작업했는지 보여주고, 결과가 마음에 안 들면 다시 생성할 수 있게 하고, 중요한 작업은 승인 후 실행하게 하는 기능들이 필요합니다.

7. 중요한 것은 PLG가 진화해야 한다는 점입니다. 결국 핵심은 사용자의 의도를 포착하고, 자율적으로 결과를 제공하며, 그 과정이 제대로 작동하고 있음을 증명하는 것입니다. 이것이 에이전트 네이티브 시대의 새로운 성장 공식으로 활용 될 수 있을 것 같습니다.
