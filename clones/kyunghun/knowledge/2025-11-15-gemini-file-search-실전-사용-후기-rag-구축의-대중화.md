Gemini File Search를 직접 써봤습니다.

삼성전자 감사보고서 527페이지 어딘가에 있는 숫자를 물었습니다. "2004년에 이익 소각으로 주식 변동이 얼마큼 있었어?" 20년 전 데이터입니다. 그것도 표 안에 들어가 있는 숫자입니다. 답변이 돌아왔습니다. "2004년 1월 15일 우선주 330,000주, 5월 4일 260,000주가 이익 소각되었습니다." 정확했습니다. 파일을 넣고 답변을 받는데까지 걸린 시간은 5분이채 되지 않습니다.

1. 예전에는 이렇게 했습니다

n8n으로 Supabase 벡터 DB를 구축해 챗봇을 만들어본 적 있습니다. Supabase에서 문서를 청크로 분할하는데 청크 크기는 얼마로 할지, 오버랩은 몇 토큰으로 할지 고민했습니다. 임베딩 모델도 선택해야 했습니다. OpenAI의 어떤 모델을 쓸까. 차원 수는 어떻게 설정할지도 따져봤습니다. n8n 워크플로우로 청크-임베딩-저장 파이프라인을 만들고, 검색할 때는 유사도 임계값과 top-k를 어떻게 설정할지 테스트했습니다. 답변 생성 프롬프트도 따로 설계했습니다. 처음이라서 꽤나 시간이 걸렸습니다. 

2. Gemini File Search는 달랐습니다

파일을 업로드했습니다. 끝입니다. 청킹 전략? 내부에서 자동으로 최적화합니다. 임베딩 모델? gemini-embedding-001로 알아서 합니다. 검색 파라미터? 시맨틱 검색으로 처리합니다. 신경 쓸 건 파일 경로 하나뿐입니다.

코드로 보면 5줄입니다.
file_search_store = client.file_search_stores.create()
client.file_search_stores.upload_to_file_search_store(
  file='report.pdf',
  file_search_store_name=file_search_store.name
)
벡터 DB 설정도, 청킹 파라미터 튜닝도 필요 없습니다.

3. 표 데이터를 어떻게 읽었을까요?

2004년 주식 변동 내역은 PDF 안의 표 형태로 존재했는데, 날짜별로 정확히 추출했습니다. 단순히 텍스트를 읽는 게 아니라, 표의 구조를 이해하고 데이터를 파싱했습니다. 527페이지 어딘가에 흩어진 정보를 찾아내고, 맥락을 이해하고, 사용자가 원하는 형태로 재구성하는 과정이 순식간에 이뤄졌습니다. "2024년에 어떤 정관변경이 있었어?"라고 물었을 때도 놀라웠습니다. 법률 문서 특유의 복잡한 문장들을 파악하고, 주요 변경사항을 구조화해서 답변했습니다. "주권의 종류 삭제 및 전자증권 관련 조문 정비", "이사회 소집 통지 기한을 24시간에서 7일로 변경" 같은 세부 내용까지 빠짐없이 정리했습니다.

왜 이게 가능했을까요? 키워드 검색과 시맨틱 검색의 차이입니다. 키워드 검색은 "정관변경"이라는 단어를 찾습니다. 시맨틱 검색은 "회사 규칙이 어떻게 바뀌었는지"라는 질문의 의도를 이해합니다. 파일을 업로드하면 자동으로 청크로 분할됩니다. 각 청크는 텍스트의 의미를 담은 숫자 벡터(임베딩)로 변환되고, gemini-embedding-001 모델을 거쳐 파일 검색 스토어에 저장됩니다. 질문을 던지면 질문도 같은 방식으로 임베딩이 되고, 가장 유사한 벡터를 가진 청크를 찾아냅니다. 청킹 전략은 기본적으로는 자동입니다. 하지만 필요하면 chunking_config로 청크당 최대 토큰 수와 오버랩을 직접 설정할 수 있습니다.

원본 파일은 48시간 후 삭제됩니다. 하지만 임베딩 데이터는 영구 보관됩니다. 파일 검색 스토어는 원본이 아니라 임베딩을 저장하는 컨테이너이기 때문입니다. 원본 527페이지 PDF는 사라져도, 그 안의 의미를 담은 벡터들은 계속 남아서 검색에 응답합니다.

4. 기술이 아니라 접근성의 문제입니다

조직에서 RAG 도입이 어려운 진짜 이유가 무엇일까요? 도메인을 가장 잘 아는 사람들이 RAG까지 구축할 여력이 없기 때문입니다. 복잡한 서류의 분석이 필요하다는 것을 압니다. 하지만 벡터 DB를 설정하고 임베딩 파이프라인을 만들 수는 없습니다. 개발팀에 요청하면 우선순위에서 밀립니다.

진입장벽이 낮아지면 더 많은 사람이 시도합니다. 재무팀의 사업보고서, 법무팀의 계약서, 인사팀의 규정집이 검색 가능한 지식 베이스가 될 수 있습니다. 개발 리소스 없이도 가능해집니다. 비용 구조도 합리적입니다. 인덱싱할 때 임베딩 비용(토큰 100만 개당 $0.15)만 내면 됩니다. 보관은 무료고, 쿼리 시간 임베딩도 무료입니다.

물론 완벽하지는 않습니다. 고도로 커스터마이징된 RAG가 필요하다면 여전히 직접 구축해야 합니다. 하지만 빠른 프로토타이핑이나 내부 문서 검색 같은 사용 사례에서는 충분해 보입니다. 복잡한 기술 보다는, 핵심 가치만 남겼습니다. "문서를 업로드하면 질문에 답한다." 이것이 됩니다.
