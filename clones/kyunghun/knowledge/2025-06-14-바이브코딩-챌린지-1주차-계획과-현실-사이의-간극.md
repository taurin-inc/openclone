바이브코딩 챌린지 1주차: 계획과 현실 사이의 간극

매주 하나씩 제품을 만들어보는 도전인데, 첫 주부터 예상과는 다소 다른 방향으로 흘러갔습니다.

1. 계획: 밈 이미지 자동 생성기
처음 아이디어는 간단했습니다. 기사 링크를 넣으면 해당 내용으로 밈 이미지를 자동 생성해주는 도구를 만드는 것이었습니다. 요즘 밈 문화가 활발하고, 특히 IT 업계에서 트렌드를 재미있게 풀어내는 밈들이 인기를 끌고 있어서 괜찮은 아이디어라고 생각했습니다.

하지만 PRD를 작성하던 중 문득 의문이 들었습니다. "이게 정말 나한테 필요한가?" 솔직히 말하면 밈을 한두 번 만들어보고는 쓰지 않을 것 같았습니다. 재미는 있지만 지속적으로 사용할 동기가 부족했습니다.

이왕이면 실제로 꾸준히 사용할 만한 것을 만들어보자. 다시 생각해보니 링크드인 썸네일 생성기가 훨씬 실용적이겠다는 생각이 들었습니다. 링크드인에 글을 올릴 때마다 썸네일을 만드는 것이 번거로웠는데, 이를 자동화할 수 있다면 정말 유용할 것 같았습니다. 방향을 전환했습니다.

2. PRD 작성의 현실과 개선
GPT에게 "이런 것 만들 거야, PRD 작성해줘"라고 대략적인 프롬프트를 주면 예상보다 훨씬 광범위한 문서가 나왔습니다. 시장 분석부터 사업전략, 수익 모델까지 포함한 문서인데 그렇게까지는 필요하지 않았습니다.

대신 유저 관점에서 접근했습니다. 이 서비스를 어떻게 이용하는지, 어떤 기능이 있고 어떤 페이지가 필요한지, 사용자가 어떤 단계를 거쳐 목표를 달성하는지에 집중해서 GPT와 작업했습니다.

예를 들어
- 사용자가 링크드인 글을 입력한다
- AI가 핵심 내용을 분석해서 콘셉트를 제안한다
- 사용자가 콘셉트를 확인하고 수정한다
- 프롬프트가 자동 생성되고 사용자가 다시 수정할 수 있다
- 최종적으로 썸네일이 생성된다
이런 식으로 단계별 사용자 여정에 집중한 PRD가 일반적인 PRD 형식은 아닐지 모르지만, 실제 개발에는 훨씬 도움이 되었습니다.

3. 도구 선택: Lovable의 진화
개발 도구를 고민했습니다. Cursor를 쓰는 것도 방법이지만, 시간도 오래 걸리고 배포도 복잡할 것 같았습니다. 이번주는 하루에 완성하려고 했기에 가장 간단한 Lovable을 선택했습니다. 마침 이번 주말에 무료로 제공되어 부담 없이 사용할 수 있었습니다.

예전에 Lovable을 써봤을 때는 너무 심플하다는 느낌이었는데, 이번에 써보니 Cursor와 비슷해진 느낌을 많이 받았습니다.
- 코드를 직접 보지 않아도 되고 웹페이지 화면만 보면서 작업할 수 있다
- 채팅으로 어디를 어떻게 수정할지 우선 대화 할수 있다
- 실시간으로 결과를 확인하면서 수정할 수 있다
- 배포가 원클릭으로 가능하다

물론 Cursor만큼 세밀한 제어는 어렵지만, 간단한 프로젝트라면 Lovable로도 충분하겠다는 생각이 들었습니다. 특히 프로토타입이나 MVP를 빠르게 만들 때는 오히려 더 효율적일 수도 있겠다고 생각했습니다.

4. 기술적 한계와 현실적 선택
가장 큰 어려움은 썸네일 품질이었습니다. 처음에는 OpenAI의 DALL-E 3를 사용했는데 이미지 퀄리티가 기대에 못 미쳤습니다. 특히 텍스트가 포함된 비즈니스용 썸네일을 만들 때 글자가 깨지거나 디자인이 아마추어스러웠습니다.

더 좋은 대안을 찾아봤습니다. Google의 Imagen4와 GPT Image-1이 최근 출시되어 품질이 훨씬 좋다는 평가를 받고 있었습니다. 이 두 모델을 활용해보려고 시간을 꽤 썼습니다.

Imagen4를 사용하려면 Google Cloud의 Vertex AI를 통해 접근해야 했습니다. API 키 발급부터 서비스 계정 설정까지 복잡한 과정을 거쳤지만, 결국 브라우저 환경에서는 직접 호출할 수 없다는 것을 알게 되었습니다. GPT Image-1도 마찬가지로 Lovable에서는 지원하지 않았습니다. 

결국 현실적인 선택은 DALL-E 3였습니다. 대신 프롬프트를 최대한 정교하게 다듬는 데 집중했습니다.

5. 프롬프트 엔지니어링의 현실: 생각보다 훨씬 어렵다
여기서 예상치 못한 가장 큰 난관에 부딪혔습니다. 프롬프트 최적화가 생각보다 훨씬 어려웠습니다.
처음에는 간단할 줄 알았습니다. "LinkedIn용 전문적인 썸네일을 만들어줘"라고 하면 적당히 괜찮은 결과가 나올 것이라고 생각했습니다. 하지만 현실은 달랐습니다.

DALL-E 3 프롬프트를 수십번 수정했습니다. “전문적이고 세련된" 같은 추상적 표현은 전혀 통하지 않았습니다. 글꼴이나 레이아웃 지시사항이 제대로 반영되지 않았습니다. 

GPT에게 프롬프트 개선을 요청해도 어느 순간부터 "이게 최선입니다"라고 하더군요. 더 이상 개선할 여지가 없다는 식이었습니다. AI 이미지 생성에서 원하는 스타일을 정확히 구현하는 것은 생각보다 훨씬 어려운 일이라는 것을 배웠습니다.

6. 핵심 레슨런
이번 1주차에서 얻은 교훈들입니다.

1) 개발은 역시 생각대로 되지 않습니다. 금방 될 것 같은 기능도 예상보다 오래 걸립니다. Lovable 같은 노코드 툴을 써도 마찬가지였습니다. 특히 AI API 통합이나 사용자 플로우 설계는 생각보다 복잡했습니다.

2) PRD의 중요성을 다시 깨달았습니다. PRD를 제대로 쓰는 게 느린 것 같아서 그냥 빨리 만들면서 해보고 싶은 마음이 들지만, 그러면 결국 다시 돌아가야 합니다. 계속 수정하다 보면 코드가 복잡해짐이 느껴집니다. 결국 몇 번의 시행착오 후에 PRD를 다시 써서 새롭게 작성하게 되었습니다.

3) 프롬프트 엔지니어링은 생각보다 훨씬 어렵습니다. 이미지 작성은 텍스트와는 또 다른 차원입니다. 원하는 스타일의 이미지를 프롬프트로 정확히 구현하는 것은 정말 어려운 것 같습니다.

완벽하지는 않지만 실제로 사용할 수 있는 링크드인 썸네일 생성기가 완성되었습니다. 아주 만족은 아니지만 그래도 쓸만은 합니다. 다음 주차에서는 어떤 새로운 도전과 배움이 기다리고 있을지 기대됩니다.
