# Amazon S3 비용 절약 및 이미지 CDN 트래픽 최적화

**작성자:** 제이크(Jake), 구피(Goofy)  
**날짜:** 2025년 10월 29일

---

안녕하세요, 인프랩 DevOps 엔지니어 제이크, 구피입니다.

인프랩은 최근 Amazon S3 보관 비용과 CDN 트래픽을 최적화하는 성과를 달성했습니다. S3 Intelligent-Tiering 스토리지 클래스 도입과, CDN 응답 이미지 AVIF·MP4 포맷 변환 지원에 관한 내용입니다.

인프랩이 무엇을 문제로 삼고 어떻게 최적화했는지, 그리고 실제로 전환하면서 고려했던 요소와 구체적인 경험을 함께 공유하고자 합니다. 자그마한 노하우지만서도 여러분께 도움이 되셨으면 좋겠습니다. 그러면 시작하겠습니다.

## Amazon S3 비용 절약하기

### 왜 비용이 많이 나오는가?

인프런은 수많은 강의 영상을 Amazon S3 버킷에 보관하고 있습니다. 강사님께서 업로드한 원본 영상과, 이를 학습자 분들께서 재생하기에 최적의 품질과 크기로 인코딩한 버전을 모두 보관합니다.

VOD 서비스의 경우 클라이언트의 네트워크 사양에 적합한 영상을 서빙할 수 있도록 동영상을 해상도별로 인코딩합니다. 주로 540p, 720p, 1080p 등 다양한 해상도에 맞게 영상을 준비해야 합니다.

오디오 트랙도 별도로 관리하고 있습니다. 간혹 볼륨이 불안정한 영상의 경우 시청할 때 큰 불편을 초래하고는 합니다. 이용자 편의를 위해 모든 오디오 트랙에 대해 볼륨 정규화(volume normalization) 단계를 VOD 파이프라인에서 거치고 있습니다.

![speech normalization](https://tech.inflab.com/static/6c23dc7b8f955fdc4932cc586dd1c834/37523/speech-normalization.png)

최근에는 재생 편의를 위해 각 구간별 프레임을 미리 볼 수 있는 썸네일 스프라이트 이미지도 VOD 파이프라인에서 처리하면서 기존 대비 훨씬 많은 객체를 관리하게 되었습니다.

![thumbnail sprite](https://tech.inflab.com/static/e67b616231a163924f3a6b626f13b072/37523/thumpnail-sprite.png)

VOD 파이프라인에서 처리하는 데이터와 지원하는 기능이 많아질수록, 여러 단계를 거치다 보니 각 단계에서 수 많은 파일이 생성됩니다. CDN 액세스 로그와 같이 대량의 로그 또한 S3 버킷에 꾸준히 적재되고 있어서, 날이 갈 수록 S3 보관 비용이 눈에 띄게 증가했습니다. 사용자 경험에 큰 영향을 주지 않으면서도 S3 보관 비용을 줄일 수 있는 방법을 모색하게 되었습니다.

### S3 스토리지 클래스

Amazon S3는 여러 종류의 스토리지 클래스를 지원합니다. **보관 비용과 조회 시간을 트레이드 오프** 할 수 있는 옵션을 제공하는 거죠.

주로 자주 사용하는 스토리지 클래스는 다음과 같습니다.

- `Standard`: 일반적인 데이터, 즉시 접근 필요
- `Intelligent-Tiering`: 접근 패턴에 따라 자동 최적화
- `Glacier`: 장기 보관, 복구(Retrieved) 시간 필요 (분~시간)
- `Deep Archive`: 가장 저렴, 복구(Retrieved) 시간 매우 긺 (시간~일)

#### S3 Lifecycle configuration

Amazon S3 서비스는 객체 관리를 자동화할 수 있도록 Lifecycle configuration이라는 기능을 제공합니다.

- 객체 업로드 시점으로부터 n일 경과시 스토리지 클래스 변경
- 이전 버전 객체의 스토리지 클래스 변경
- n일 경과시 객체 만료(expire), 즉 영구삭제

정말 유용한 기능이지만, 객체의 스토리지 클래스를 변경할 때에는 API 호출 비용이 부과되며, 객체 생성 일자 단위로만 클래스를 변경할 수 있어서 자주 접근하는 `hot`, 그리고 가끔 접근하는 `cold` 객체를 서로 구분하여 관리하기 어렵습니다.

#### 어려운 이유

실제 서비스를 운영하다보면, 객체 접근 패턴은 예측이 매우 어렵습니다. 몇몇 스토리지 클래스에서는 최소 보관 기간을 두는데, 이 기간이 지나기 전에 조회하면 추가 비용이 발생합니다. 또한 구간별로 스토리지 클래스를 전환할 때마다 추가 비용이 발생합니다.

인프랩 DevOps 팀에서도 이러한 이유로 비용 최적화에 엄두를 내지 못하고 있었으나, 최근 스토리지 비용 부담이 커지게 되어 Intelligent-Tiering 도입을 결정하게 되었습니다.

### S3 Intelligent-Tiering

다음 조건을 충족한다면 S3 Intelligent-Tiering 도입을 적극 권장합니다.

1. 객체 크기가 **대부분 128KB를 넘는 편**이다.
2. 콜드 스토리지임에도 **모든 객체가 수 밀리 초 안에 조회가 가능해야** 한다.
3. 별도의 캐싱 계층(CDN 등)이 있다.
4. 객체 단위의 자동 관리가 필요하다.

Intelligent-Tiering은 파일 개수 만큼 관리 요금만 지불하면 클래스 전환 비용이나 복구 비용 없이 **자동으로 보관 비용을 절약**해줍니다.

- `IA` (Infrequent Access): Standard 대비 44.8% 저렴
- `AIA` (Archive Instant Access): Standard 대비 80% 저렴

#### 도입 비용

1. **객체 관리 비용:** 1000개의 객체 당 $0.0025
2. **Lifecycle 전환 요청 비용:** 1000개의 객체 당 $0.01

#### CopyObject를 직접 호출하는 방법

Lifecycle 규칙 대신 `CopyObject` API를 직접 사용하면 전환 요청 비용을 약 2.2배 저렴하게 처리할 수 있습니다. 같은 리전 내 전송은 무료이므로 객체 수가 많을 경우 수동 전환 방식도 고려할 만합니다.

### 진행 및 성과

1. 신규 업로드 객체에 Intelligent-tiering 즉시 적용.
2. 아카이브 버킷은 `CopyObject`로 일괄 전환.
3. 인코딩 버킷은 Lifecycle rule을 사용해 점진적 적용.

적용 후 3~4달이 지나 자주 접근되지 않은 객체의 비용이 감소하여 **전체 S3 비용의 38%를 절감**할 수 있었습니다.

![s3 비용 절감](https://tech.inflab.com/static/a2d273f417e4228b52c450b880c7be64/37523/s3-cost.png)

#### S3 Inventory 활용

도입 전 S3 Inventory를 활용해 객체 크기 분포와 예상 전환 비용을 사전에 산정했습니다. Athena 쿼리를 통해 128KB 초과 파일 개수를 확인하는 등의 분석을 거쳐 매끄럽게 전환할 수 있었습니다.

## 이미지 CDN 트래픽 최적화

### 왜 비용이 많이 나오는가?

인프런은 메인 페이지 등 거의 모든 곳에서 강의 썸네일을 보여줍니다. 이미지가 많을수록 로딩 시간이 길어지고 CDN 트래픽 비용도 함께 증가합니다. 이미지 최적화는 비용 절감과 사용자 경험 개선을 위해 필수적이었습니다.

### Lambda@Edge 이미지 리사이저

CloudFront의 엣지 로케이션에서 실행되는 Lambda@Edge를 사용하여 요청 시점에 이미지를 리사이징하고 포맷을 변환하는 on-the-fly 방식을 사용했습니다.

#### AVIF 변환 추가

AVIF 포맷은 WebP보다 20~30% 더 작은 파일 크기로 동일한 품질을 제공합니다.

| 포맷 | 파일 크기 | 절감율 |
| --- | --- | --- |
| 원본 JPG | 100 KB | - |
| WebP | 68 KB | 32% |
| AVIF | 55 KB | 45% |

Sharp 라이브러리를 사용하여 AVIF 변환을 추가했습니다. `?w=960&f=avif` 형태로 요청 시 변환된 이미지를 제공합니다.

#### 브라우저 지원

AVIF를 지원하지 않는 환경을 위해 `<picture>` 태그로 폴백(fallback)을 구성했습니다. AVIF -> WebP -> 원본 순으로 브라우저가 선택하도록 하여 호환성을 유지했습니다.

### GIF 트래픽 최적화

애니메이션 GIF는 파일 크기가 매우 큽니다. 분석 결과 AVIF(animated)나 MP4(H.264)로 변환 시 용량을 획기적으로 줄일 수 있었습니다.

| 포맷 | 파일 크기 | 절감율 |
| --- | --- | --- |
| 원본 GIF | 1000 KB | - |
| WebP (animated) | 860 KB | 14% |
| AVIF (animated) | 281 KB | 72% |
| MP4 (H.264) | 322 KB | 68% |

#### MP4 변환 람다

Lambda@Edge의 제약(패키지 크기 50MB, 실행 시간 등) 때문에 GIF 변환은 업로드 시점에 일반 Lambda와 FFmpeg를 사용하여 미리 MP4로 변환해두는 방식을 선택했습니다. 프론트엔드에서는 MP4가 존재하면 `<video>` 태그를 통해 자동 재생합니다.

### 진행 및 성과

트래픽이 높은 주요 페이지부터 단계적으로 적용했습니다. AVIF 포맷 적용 후 이미지 응답 크기(P90)가 평균 **약 15% 감소**했으며, 랠릿 서비스에서는 LCP 지표가 **1.16초에서 0.87초로 개선**되었습니다.

![이미지 CDN 요청당 응답 크기 P90](https://tech.inflab.com/static/20793453428fcfaa97b43e3277e2a89c/37523/cloudfront-req-size-avg.png)

### 모니터링

DataDog RUM과 Athena + Grafana 대시보드를 통해 리소스 크기와 페이지 로딩 속도를 지속적으로 모니터링하며 최적화 효과를 점검하고 있습니다.

## 마치며

인프랩의 S3 비용 및 CDN 트래픽 최적화 요약:

1. **S3 Intelligent-Tiering:** 복잡한 설정 없이 자동 관리로 **38% 비용 절감**.
2. **이미지 최적화:** Lambda@Edge와 AVIF 도입으로 **응답 크기 15% 개선**.
3. **GIF 최적화:** MP4 사전 변환을 통해 애니메이션 용량 대폭 축소.

작은 부분부터 점진적으로 확대하며 적용한 결과 부담 없이 성과를 낼 수 있었습니다. 이 사례가 여러분의 서비스 최적화에 도움이 되길 바랍니다. 감사합니다.
