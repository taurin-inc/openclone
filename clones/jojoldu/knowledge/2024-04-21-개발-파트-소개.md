# 개발 파트 소개 - 1. 백엔드 파트

**Author:** 향로(HyangLo)
**Date:** 2024-04-22

---

안녕하세요.

인프랩의 향로입니다.

저희 인프랩 개발팀은 현재 백엔드, 프론트엔드, 모바일, 데브옵스 파트로 나뉘어져 있습니다.

각각의 개발 파트가 어떻게 일 하는지, 어떤 사람과 함께 하고 싶어하는지 등을 소개드리려 합니다.

첫 번째로 백엔드 파트를 소개드리겠습니다.

지원하고자 하는 회사의 백엔드 개발자 채용 공고를 본다면 일반적으로 다음과 같은 궁금증이 생깁니다.

- 같이 일하게 될 사람들은 어떤 사람들일까?
- 배포 프로세스는 어떻게 될까? 자동화가 되어있을까?
- 코드리뷰는 하고 있는걸까? 서비스 기능 개발하기에 바빠서 각개전투 하는건 아닐까?
- 백엔드 역량 향상이 가능할 정도의 **유의미한 트래픽이 발생하고 있을까**?
- 테스트 코드를 작성하고 있을까?
- 모니터링과 로깅은 제대로 구축되어있을까?
- 구성원의 성장을 위해 회사에서는 어떤 지원을 해줄까?

개발자의 성장에 있어 가장 중요한건 개인의 노력인데요.

그렇다면 회사에서 해줄 수 있는 것은 무엇일까요?

제 생각에는 **그 사람이 성장할 수 있는 좋은 재료와 환경** 을 제공해주는 것이 아닐까 싶습니다.

지금부터는 인프랩의 백엔드 파트를 소개드리려고 합니다.

이런 환경이라면 **좋은 개발자로 성장할 수 있지 않을까 생각** 해보는 시간이 되셨으면 합니다.

# 직군 소개

현재 인프랩 백엔드 팀은 1년차부터 10년차까지 총 **8명의 개발자** 로 이루어져있습니다.

이 8명의 인프랩 백엔드 개발자는 **24시간 365일 장애 없는 서비스 환경** 구성과 **데이터 기반의 비즈니스 로직** 개발을 담당합니다.

![intro](https://tech.inflab.com/static/f8424cf7675d875ad9d0c59d350e1896/a4d88/intro.png)

- NodeJS 기반으로 백엔드 비지니스 로직을 개발하고 배포합니다.
- 데이터 성격에 맞게 PostgreSQL과 Redis, MongoDB Atlas에 데이터를 적재하고 이용하는 코드를 작성합니다.
- SEO를 위한 Server Rendering과 Client Rendering을 위한 API를 설계하고 개발합니다.
- 여러 백엔드 개발자들이 **협업하기 좋은 코드** 로 지속적으로 리팩토링합니다.
- **테스트 코드** 기반의 개발을 진행하며, 타입지향, 정적분석등 같이 협업하는 분들의 심리적 안정감을 주기 위해 노력합니다.
- 온라인 PR, 오프라인 페어프로그래밍 등 코드리뷰를 통해 성장하는 문화를 지향하고 있습니다.
- 데이터베이스의 실행계획 (`Explain`) 을 통한 인덱스 설계/수정과 슬로우 쿼리에 대한 튜닝 등을 진행합니다.
- 모니터링 서비스인 Datadog 을 적극적으로 사용하여 시스템에서 발생하는 모든 이슈 사항에 대해서 **추적하고 해결합니다**.

이외에 인프런 백엔드팀의 개발자들은 어떤 고민들을 하고 있고, 어떤 문제들을 해결하기 위해 노력하고 있는지는 기술 블로그를 통해 확인할 수 있습니다.

# 기술 스택

현재 백엔드 기술 스택은 크게 2가지로 나눠져있습니다.

- 현재 대부분의 인프런 기능을 담당하고 있는 기존 시스템
- 새 기술 스택으로 기존 기능을 하나씩 이관 중인 신규 시스템

각각의 기술 스택은 목적에 맞게 다르게 구성되어 있습니다.

## 기존 시스템 스택

- NodeJS / JavaScript
- Express
- FxJS (함수형 라이브러리)
- FxSQL (SQL Builder)
- Date-fns
- AWS Lambda를 통한 스케줄링 & 배치 환경
- Circle CI를 통한 배포 환경
- FP 로만 이루어진 애플리케이션 설계와 구조

## 신규 시스템 스택

- NodeJS / TypeScript
- Nest.js (Mono Repo / Multi Module 사용)
- TypeORM / MikroORM
- Jest / SuperTest
- AWS LocalStack (AWS 서비스를 로컬 테스트 하기 위해)
- js-joda / ts-jenum
- Docker Compose를 통한 개인 개발 환경
- Jenkins를 통한 스케줄링 & 배치 환경
- Jenkins / Github Action 를 통한 배포 환경
- OOP / 도메인 기반의 애플리케이션 설계와 구조 (FP가 필요한 영역과 OOP가 필요한 영역을 조화롭게 사용)

이외 인프라 요소는 두 시스템 모두 비슷한 형태를 취하고 있습니다.

- Docker
- DataDog 모니터링
- AWS Aurora PostgreSQL, Elastic Cache Redis, MongoDB Atlas
- AWS ECS (Fargate, EC2) & EC2
- Git/GitHub
- Webstorm / VSCode / DataGrip (JetBrains사의 전제품 지원)

신규 시스템의 기술 스택은 신규 입사자의 러닝커브, IDE 지원, 타입 안정성, 개발자 시장 Pool, 라이브러리 호환성 등을 고려하여 선택되었습니다. 기존의 기술 스택을 신규 기술스택으로 전환하는 과정을 제대로 경험해보실 수 있습니다.

# 업무 방식

스타트업 특유의 **빠른 제품 릴리즈** 와 **서비스 안정성** 이라는 2마리 토끼를 최소한의 인원으로 잡기 위해 적절히 업무 균형을 유지하고 있습니다.

- **Confluence / Jira** 를 도입하여 문서화를 적극적으로 진행하고 있습니다.
  - 특정 개인에게 의존하는 위험도를 낮추고, 신규 입사자분들의 심리적 안정감을 위해 개발파트 전체가 문서화에 적극 참여합니다.
  - 장애 내역과 해결 사례를 남겨 전체 구성원의 역량을 끌어올립니다.

- **정기적인 스프린트와 회고**
  - Cell(스쿼드) 단위 2주 스프린트, 개발 조직 전체 주 1회 공유 세션 진행.
  - 업무 방식 자체에 대해서도 주기적으로 회고하며 효율성을 고민합니다.

- **코드 퀄리티 및 배포**
  - Git Flow 기반 브랜치 관리 및 코드리뷰 진행.
  - 소나큐브(SonarQube) 연동을 통한 코드 퀄리티 보장.
  - 신규 프로젝트 테스트 코드 작성 필수 및 최소 커버리지 관리.
  - Jenkins를 통한 ECS 자동 배포 환경 구축.

- **모니터링 및 대응**
  - Slow Query 슬랙 알람 및 쿼리 튜닝.
  - DataDog Log와 APM을 통한 서비스 장애 대응.

- **반기별 제품 개발 파트 전체 회고**
  - 진행 과제 소개 및 개선점 논의를 통해 방향성을 점검합니다.

# 백엔드 개발자로서 얻을 수 있는 것들

- **다양한 도메인 경험**
  - 강의, 스트리밍, 채용, 커뮤니티, 결제/정산 등 백엔드 개발자가 익혀야 할 전반적인 도메인을 모두 다룹니다.
  - 대형 프로젝트와 분산 서비스를 동시에 경험할 수 있습니다.

- **안정적으로 성장하는 서비스 경험**
  - 매년 50~300% 성장 중인 서비스에서 기술 부채와 사업적 성장의 균형을 맞추는 법을 배웁니다.

- **레거시 개편 경험**
  - 안정화된 서비스에서 레거시를 현대적인 구조로 개편하는 희귀한 경험을 할 수 있습니다.

- **함께 공부하는 문화**
  - 매주 수요일 '테크 리뷰' 시간을 통한 기술 공유 및 학습.

# 마무리

일단 실행되는 백엔드 애플리케이션을 만드는 일은 쉽습니다. 그러나 **협업하기에 좋은 방식으로, 성능과 안정성까지 고려한 백엔드 애플리케이션을 만드는 개발** 은 쉽지 않습니다.

모니터링, 로그관리, 테스트하기 좋은 구조 등 운영 레벨에서 중요한 점들이 백엔드에서는 큰 비중을 차지합니다. 현재 인프랩은 서비스가 성장함에 따라 기존 방식이 더이상 효율적이지 않다는 것을 매일 경험하고 있습니다. 이런 성장의 시기에만 느낄 수 있는 경험을 같이 해보았으면 합니다.

긴 글 끝까지 읽어주셔서 감사합니다.
