## 본문

빌드툴은 Gradle을 사용할 예정입니다. IDE는 IntelliJ 를 사용합니다.

build.gradle은 로그 관리를 위해 2개의 의존성을 먼저 추가하고 시작하겠습니다.

**build.gradle**
```gradle
dependencies {
    compile group: 'org.slf4j', name: 'slf4j-api', version: '1.7.25' // log를 위해 slf4j 추가
    compile group: 'ch.qos.logback', name: 'logback-classic', version: '1.2.3' // log를 위해 logback 추가
}
```

기본적인 디렉토리 구조는 `src/main/java,resources` & `src/test/java,resources`이며 기본 패키지는 `myjunit`입니다.

### 1. 첫 테스트 코드

간단하게 테스트 코드를 작성하겠습니다. `src/test/java/myjunit`에 `TestCaseTest.java`를 생성합니다.

```java
public class TestCaseTest {

    public static void main(String[] args) {
        new TestCaseTest().runTest();
    }

    public void runTest () {
        long sum = 10+10;
        Assert.assertTrue(sum == 20);
    }
}
```

이 테스트 코드를 수행하기 위해 `Assert` 클래스와 static Method인 `assertTrue`를 생성하겠습니다.

```java
public class Assert {
    private static final Logger logger = LoggerFactory.getLogger(Assert.class);

    private Assert() {} // 인스턴스 생성을 막기 위해 기본생성자 private 선언

    public static void assertTrue(boolean condition) {
        if(!condition){
            throw new AssertionFailedError();
        }

        logger.info("Test Passed");
    }
}
```

`AssertionFailedError` 클래스도 추가 생성합니다.

```java
public class AssertionFailedError extends Error {
    public AssertionFailedError() {}
}
```

이렇게 구성하고 `TestCaseTest`의 main 메소드를 실행하면 첫 번째 테스트가 성공함을 확인할 수 있습니다.

### 2. TestCase

현재 구조는 테스트 프레임워크라고 할 수는 없습니다. 각각의 테스트 케이스 단위로 요청을 나눌 수 있는 구조가 되어야 합니다. 이럴 때 가장 어울리는 것이 **커맨드 패턴**입니다.

**TestCase.java**
```java
public abstract class TestCase {

    protected String testCaseName;

    public TestCase(String testCaseName) {
        this.testCaseName = testCaseName;
    }

    public abstract void run();
}
```

위 클래스를 사용하여 `TestCaseTest` 코드를 수정합니다.

```java
public class TestCaseTest extends TestCase {

    public TestCaseTest(String testCaseName) {
        super(testCaseName);
    }

    @Override
    public void run() {
        try {
            Method method = this.getClass().getMethod(super.testCaseName, null);
            method.invoke(this, null);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    public void runTest() {
        long sum = 10+10;
        Assert.assertTrue(sum == 20);
    }

    public static void main(String[] args) {
        new TestCaseTest("runTest").run();
    }
}
```

각 테스트가 독립적으로 실행되도록 모든 테스트 케이스는 새로운 인스턴스에서 수행됩니다. 여기서 오버라이딩하고 있는 `run` 메소드는 공통된 로직이므로 부모인 `TestCase`로 이동시킵니다.

**리팩토링된 TestCase.java**
```java
public abstract class TestCase {
    private static final Logger logger = LoggerFactory.getLogger(TestCase.class);
    protected String testCaseName;

    public TestCase(String testCaseName) {
        this.testCaseName = testCaseName;
    }

    public void run() {
        try {
            logger.info("{} execute ", testCaseName);
            Method method = this.getClass().getMethod(testCaseName, null);
            method.invoke(this, null);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
```

### 3. Fixture 메소드

`@Before`, `@After`와 같이 공통적으로 수행되는 메소드(Fixture) 기능을 추가하겠습니다. 이는 **템플릿 메소드 패턴**을 적용하기에 적절합니다.

```java
public abstract class TestCase {
    ...
    public void run(){
        before();
        runTestCase();
        after();
    }

    protected void before() {}

    private void runTestCase() {
        try {
            logger.info("{} execute ", testCaseName);
            Method method = this.getClass().getMethod(testCaseName, null);
            method.invoke(this, null);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    protected void after() {}
}
```

이제 테스트 클래스에서 `before()`를 오버라이딩하여 공통 자원을 설정할 수 있습니다.

### 4. TestResult

전체 테스트 중 몇 개가 성공하고 실패했는지 레포팅하기 위해 **Collecting Parameter 패턴**을 사용합니다.

**TestResult.java**
```java
public class TestResult {
    private static final Logger logger = LoggerFactory.getLogger(TestResult.class);
    private int runTestCount;

    public TestResult() {
        this.runTestCount = 0;
    }

    public synchronized void startTest() {
        this.runTestCount++;
    }

    public void printCount(){
        logger.info("Total Test Count: {}", runTestCount);
    }
}
```

`TestCase` 클래스의 `run` 메소드가 이 `TestResult`를 받도록 수정합니다.

### 5. Fail 처리

테스트가 하나 실패하더라도 다른 테스트는 계속 진행되어야 하며, 실패(Failure)와 오류(Error)를 구분해야 합니다.

**TestCase.java**
```java
public void run(TestResult testResult){
    testResult.startTest();
    before();
    try{
        runTestCase();
    } catch (InvocationTargetException ite) {
        if(isAssertionFailed(ite)){
            testResult.addFailure(this);
        } else {
            testResult.addError(this, ite);
        }
    } catch (Exception e) {
        testResult.addError(this, e);
    } finally {
        after();
    }
}
```

`TestResult`에는 실패와 오류 리스트를 관리하는 로직과 레포팅 로직을 추가합니다.

### 6. TestSuite

개별 테스트들을 그룹화하여 실행하기 위해 **Composite 패턴**을 도입합니다.

**Test.java (Component)**
```java
public interface Test {
    void run(TestResult result);
}
```

**TestSuite.java (Composite)**
```java
public class TestSuite implements Test {
    private List<TestCase> testCases = new ArrayList<>();

    @Override
    public void run(TestResult result) {
        for (TestCase testCase : this.testCases) {
            testCase.run(result);
        }
    }

    public void addTestCase(TestCase testCase) {
        this.testCases.add(testCase);
    }
}
```

이제 `TestCase`도 `Test` 인터페이스를 구현하게 하여, 개별 테스트와 그룹 테스트를 동일한 방식으로 다룰 수 있게 됩니다.

### 7. 정리

JUnit 프레임워크를 직접 만들어보며 Command, Template Method, Collecting Parameter, Composite 패턴 등 다양한 디자인 패턴을 적용해 보았습니다.

객체지향과 디자인 패턴을 이해하는 데 이 방식은 큰 도움이 됩니다. 시간이 되신다면 여러분도 직접 구현해 보시길 추천드립니다.
