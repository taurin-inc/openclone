최근 너무 재밌게 본 기술 서적이 2권 있는데, 현석님이 번역해주신 "객체에서 함수로" 와 성철님이 번역 해주신 "데이터 지향 프로그래밍" 이다.
OOP 로 대부분의 문제를 풀었던 나 같은 사람은 이런 '객체' 가 아닌 '함수' 혹은 '데이터' 중심으로 사고를 전환하는 것이 어렵다.
와닿지 않고, 이게 또 다른 새로운 해결책 중 하나라는 것을 인정하기 어렵다.
그러다보니 이런 책들을 잘 안보게 되는데, 그럼에도 이런 컨텐츠를 통해 개발자로서 성장하는 부분이 분명히 있다.

데이터 지향 프로그래밍 (DOP)은 함수형 프로그래밍 (FP) 과는 비슷하지만 결이 다르다.
그럼에도 기존의 OOP가 추구하는 가치와는 많이 다르다.
특히 Java 17부터 지원되는 Record 는 OOP 보다는 FP, DOP에 훨씬 더 적합한 도구다.

예전 Java (종종 Groovy)를 주로 사용할 때는 Java 8~11 환경에서만 사용하다보니 함수형 프로그래밍에 대해서 크게 관심이 없었다.
stream 기반의 map, filter를 사용하거나 Functional Interface 기반으로 함수를 인자로 전달하는 등 "이런 단편적인 기능을 사용한다고 해서 함수형 프로그래밍을 한다고 할 수는 없다" 정도는 인식하고 있었지만, 그 이상에 대해서는 관심이 없었다.

이후에 지금 조직에 합류하면서 전반적으로 FP를 좋아하는 성향의 팀 분들을 만나고 JS, TS를 적극적으로 사용하게 되면서 왜 FP를 좋아하시는지 조금씩 느낄 수 있었다.
백엔드가 OOP에 적합하다곤 하지만 실제로 외부와 통신하는 등의 일부 계층에서는 오히려 FP가 훨씬 더 좋다는 것도 깨닫게 되었다.

OOP는 복잡한 문제를 구조적으로 잘 쪼개어 해결하기 편리하지만, 간단한 문제마저도 불필요하게 복잡해지게 만들었다.
반면 FP는 상태나 부수효과를 최소화해서 문제를 간결하게 해결하지만, 시스템의 요구사항이 복잡하여 상태 변화가 여러 객체/모듈에 걸쳐 발생한다면 복잡성을 오히려 제어하기 어려웠다.

이런 점들이 특정 언어 하나만 하거나 특정 프로그래밍 패러다임 하나에만 매몰되어 있을 때는 몰랐다.
모든걸 객체로 표현해도 문제를 해결하는데 어렵지 않았으니깐.
근데 객체로 모든 문제를 풀 수는 있지만,
문제를 단순화 시켜주고, 더 쉽게 풀어주지 못할 때가 있다.

그래서 어떤 문제를 만났을때 여러 형태로 풀어보는 경험이 대단히 중요하다.
그래야 적절한 도구를 선택할 수 있다.

그동안 OOP에 대해서 방대한 자료들이 주변에서 쉽게 찾아볼 수 있었다.
특히 국내 저서로 이미 영호님의 토끼책과 오브젝트 등 좋은 OOP 책들이 너무나 많기 때문에 OOP를 잘 배울 수 있는 방법은 많았다.

헌데, OOP와 FP 양 쪽을 모두 다 다루면서 둘은 어떤것을 지향하고 어떤 점이 다른지 자세히 알려주는 국내 컨텐츠가 거의 없었다.
(스칼라나 그루비는 요즘 선호되지 않는 언어이기도 하고)

다행히도 이번에 마플의 CTO이신 유인동 님이 여러 패러다임에 대해 다루시는 멀티 패러다임에 관한 강의와 책을 내주셨다.
국내에서도 이런 컨텐츠가 나올 수 있다는 게 대단히 좋았다.
프론트엔드는 FP
백엔드는 OOP
대부분의 국내 개발이 규칙처럼 정해진 채로 개발을 하고 있는데, 이에 대해 많은 환기를 시켜줄 수 있는 컨텐츠라고 생각한다.

인동님의 컨텐츠가 이번 편 이후에도 계속 이어질텐데,
앞으로의 컨텐츠가 더더욱 기대된다.
