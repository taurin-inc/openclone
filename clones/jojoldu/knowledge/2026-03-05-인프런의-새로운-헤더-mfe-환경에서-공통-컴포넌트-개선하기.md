# 인프런의 새로운 헤더: MFE 환경에서 공통 컴포넌트 개선하기

**작성자:** 고슈 (GOHSUE)  
**날짜:** 2026년 3월 5일

---

안녕하세요. 인프랩 프론트엔드 개발자 고슈입니다.

_**작년 8월, 인프런의 헤더(GNB)가 새롭게 단장했습니다! 🎉**_

인프런의 헤더는 여러 프론트엔드 환경에서 동작해야 하는 복잡한 구조로 이루어져 있어, 기존에도 여러 문제가 있었는데요.

이번 글에서는 인프런의 헤더가 어떻게 동작하고 있는지, 그리고 어떤 과정을 거치며 문제를 해결해 나갔는지를 담아 보았습니다.

## 1. 헤더를 왜 개편하게 되었는가

![기존 헤더](https://tech.inflab.com/static/476570d96d8527c1740c46e58158e913/37523/header-as-is.png)
*기존 헤더*

![개편된 헤더](https://tech.inflab.com/static/6af49e6b2108a2e92dbe32a785f05659/37523/header-to-be.png)
*개편된 헤더*

인프런의 헤더는 런칭 이래로 변경된 적이 거의 없었는데요. 서비스가 추가될 때마다 기존 구조에 끼워 넣기만 하다 보니, 핵심 서비스가 명확하게 드러나지 않는 상태였습니다.

현재 인프런은 **강의, 챌린지, 멘토링, 클립, 커뮤니티**를 핵심 서비스로 정의하고 있습니다. 이번 프로젝트의 목표는 이 핵심 서비스들을 헤더에서 명확하게 노출하고, 사용자가 직관적으로 탐색·검색·구매·학습할 수 있도록 개선하는 것이었습니다.

디자인 개편 자체는 이미 완료되었지만, 그 과정이 단순하지만은 않았는데요. 이어지는 내용에서는 이 헤더가 동작하고 있는 MFE 환경의 구조와, 그 안에서 마주한 기술적 문제들을 하나씩 풀어 보겠습니다.

---

## 2. MFE 아키텍처와 Module Federation

### 2-1. 인프랩 팀의 프론트엔드 아키텍처

![모놀리식 아키텍처에서 도메인 별로 프로젝트 분리](https://tech.inflab.com/static/54e6f9488a43b602bd21f1b7dea848df/37523/diagram-monolith.png)

#### 모놀리식에서 도메인 분리로
초기 인프런은 내부에서 Antman이라 부르는 자체 자바스크립트 프레임워크 기반의 모놀리식 프로젝트로 서비스를 운영하고 있었습니다. 서비스 규모가 커지면서 개발자 수도 늘고 빌드 속도도 느려졌으며, 무엇보다 프론트엔드와 백엔드가 함께 있어 한쪽의 장애가 서비스 전체 마비로 이어지는 문제가 있었습니다.

이를 해결하기 위해 리액트로 마이그레이션을 진행하며 도메인별로 프로젝트를 분리했습니다. 각 도메인이 독립적으로 운영되는 마이크로프론트엔드(MFE) 구조이며, 리버스 프록시를 통해 하나의 서비스처럼 동작합니다. 이를 통해 독립적인 배포와 장애 격리가 가능해졌습니다.

#### 분리했더니, 더 바빠졌다
하지만 헤더와 같은 공통 컴포넌트를 수정할 때마다 모든 패키지에 반영하고 배포해야 하는 번거로움이 생겼습니다. 패키지가 10개라면 10번 배포해야 하는 셈이었죠. 이를 해결하기 위해 세 가지 방법을 검토했습니다.

1.  **서버 사이드 컴포지션**: 프론트엔드 전용 서버에서 결과물을 조합하는 방식. 서버 장애 시 모든 페이지 마비 위험이 큼.
2.  **CI/CD 디스패치 자동 배포**: 변경 시 각 패키지에 자동 배포 명령을 보냄. 동료의 작업과 충돌하거나 배포 결과를 일일이 확인해야 하는 불편함이 있음.
3.  **런타임 로딩 (Module Federation)**: 번들링된 JS를 S3에 올리고 각 앱이 런타임에 가져오는 방식. 헤더만 배포하면 즉시 반영되어 유지보수가 용이함.

저희는 독립적인 환경에서의 개발자 경험(DX)을 위해 **세 번째 방식인 Module Federation**을 채택했습니다.

## 3. 기존 헤더의 한계

### 3-1. 이중 관리 방식
Module Federation을 도입했지만, 여전히 리액트로 마이그레이션되지 않은 레거시 페이지(Antman)가 문제였습니다. Antman용 헤더(순수 JS)와 앱쉘용 헤더(리액트)를 이중으로 관리해야 하는 상황이었습니다.

#### 해결: 어댑터 모듈을 이용한 단일 관리
리액트 컴포넌트를 JS로 번들링하여 S3에 올리고, Antman에서 런타임에 로드하는 '어댑터 모듈' 방식을 헤더에 적용했습니다.

*   **번들링 호환**: Vite를 이용해 Antman 전용 엔트리 포인트를 구성하여 단독 실행 가능한 번들을 생성했습니다.
*   **스타일 충돌**: `emotion`을 사용해 class name을 난수화하고, Antman의 글로벌 CSS와 겹치지 않도록 처리했습니다.
*   **Layout Shift**: `MutationObserver`를 활용해 헤더 로드 전 placeholder를 배치하고, DOM 추가 시점에 교체하여 레이아웃 밀림을 방지했습니다.

### 3-2. 런타임 로딩으로 인한 깜빡임 문제
런타임에 JS를 불러오는 방식이라 페이지 이동 시마다 스켈레톤 UI가 노출되며 깜빡이는 현상이 발생했습니다.

#### 해결: 스냅샷 접근법
이전에 렌더링된 헤더 상태를 쿠키에 저장해 두고, 서버에서 HTML을 내려줄 때 이를 활용해 정교한 스켈레톤을 먼저 보여주는 방식입니다.

1.  **쿠키 저장**: 헤더의 마지막 상태(메뉴 목록 등)를 쿠키에 저장합니다.
2.  **data attribute + CSS 제어**: 하이드레이션 에러를 방지하기 위해 컴포넌트 트리를 수정하지 않고, `<html>` 태그의 `data attribute`와 `CSS 변수`를 통해 스켈레톤의 노출 여부와 텍스트를 제어했습니다.

이 방식을 통해 SSR로 전환하지 않고도 사용자에게는 깜빡임 없는 자연스러운 경험을 제공할 수 있었습니다.

---

## 4. UX 요구사항 해결: 자연스러운 레이아웃과 모션

### 4-1. 경로별 검색바 영역 확보
경로에 따라 검색바 노출 여부가 달라지는 문제를 스냅샷 방식과 동일하게 `data attribute`로 해결했습니다. 서버 응답 시점에 경로 정보를 바탕으로 검색바 공간을 미리 확보하여 Layout Shift를 방지했습니다.

### 4-2. 스크롤 모션의 성능 개선
검색바가 스크롤 시 축소되는 애니메이션을 처음에는 `height` 변경(Reflow 유발)으로 구현했으나, 요소가 많은 페이지에서 버벅임이 발생했습니다.

이를 해결하기 위해 **GPU 가속을 활용하는 `transform: translateY()`** 방식으로 변경했습니다. Reflow 없이 Composite 단계에서 처리되도록 구조를 개선하여 리스트 페이지에서도 자연스러운 모션을 구현했습니다.

---

## 5. 완벽한 구조 대신, 현실적인 구조를 선택하기까지

가장 깔끔한 해결책은 SSR 직접 렌더링이었겠지만, 저희는 DX(개발자 경험)와 운영 효율성을 위해 Module Federation과 스냅샷이라는 우회 전략을 선택했습니다. 헤더만 독립적으로 배포할 수 있는 장점을 유지하면서 UX 문제를 수용 가능한 수준으로 끌어올린 것입니다.

| 구분 | 개편 전 | 개편 후 |
| --- | --- | --- |
| 헤더 코드 관리 | 앱쉘 + Antman 두 벌 | 앱쉘 한 벌로 통합 |
| 헤더 변경 시 배포 | 모든 패키지 개별 배포 | 앱쉘만 배포 |
| 스켈레톤 | 실제 헤더와 불일치 | 스냅샷 기반 정교한 스켈레톤 |
| 스크롤 모션 | reflow 기반 (CPU) | Composite 기반 (GPU) |

---

## 6. 아직 남아 있는 과제

*   **앱쉘 API 로딩 속도**: 시각적 깜빡임은 줄였으나, 사용자 정보 등 API 응답 지연을 줄이는 과제가 남아 있습니다.
*   **Antman 마이그레이션**: 레거시 페이지들이 모두 리액트로 전환되면 어댑터 레이어를 제거하고 구조를 더 단순화할 예정입니다.
